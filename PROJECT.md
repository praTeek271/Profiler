# Profiler — Project Specification

## 1. Project Overview

**Project Name:** Profiler

Profiler is a small-scale cloud-based healthcare profile service where patients can store basic medical information as key–value pairs. Doctors can retrieve patient information using a unique identifier called the **Social Healthcare Number (SHN)**.

The project is designed as a **simple but production-style cloud architecture** to demonstrate backend development using AWS serverless technologies.

The goal is to build a **clean, minimal system that demonstrates real-world architecture without unnecessary complexity**.

---

# 2. Project Goals

### Primary Goal

Build a working serverless backend system where:

* Patients store key-value medical data
* Doctors retrieve that data using a unique identifier
* Data retrieval is optimized with caching

### Secondary Goal

Demonstrate knowledge of:

* Serverless backend architecture
* API Gateway usage
* DynamoDB data modeling
* Redis caching
* Next.js frontend integration
* AWS deployment with Amplify

---

# 3. Core Concepts

## Social Healthcare Number (SHN)

Every patient receives a unique identifier called the **Social Healthcare Number (SHN)**.

Example:

SHN-482193

This identifier acts as the **primary lookup key** for retrieving patient data.

Rules:

* SHN is generated by the backend
* SHN must be unique
* SHN never changes after creation

# 6. Data Model

Patient data is stored as **key-value pairs**.

Example patient profile:

```
SHN: SHN-482193

blood_group : O+
age : 31
allergy : penicillin
height_cm : 178
```

# 7. DynamoDB Table Design

Table Name:

ProfilerTable

Primary Keys:

PK (Partition Key)
SK (Sort Key)

---

## Patient Data Records

Format:

```
PK = PATIENT#<SHN>
SK = DATA#<key>
```

Example records:

```
PK: PATIENT#SHN-482193
SK: DATA#blood_group
value: O+
```

```
PK: PATIENT#SHN-482193
SK: DATA#age
value: 31
```

# 8. Redis Cache Design

Cache Key Format:

```
patient:<SHN>
```

Example:

```
patient:SHN-482193
```

Cached Value Example:

```
{
  "blood_group": "O+",
  "age": 31,
  "allergy": "penicillin"
}
```

TTL (Time To Live):

300 seconds

---

# 9. Cache Strategy

When a doctor requests patient data:

1. Check Redis cache
2. If cache HIT → return cached data
3. If cache MISS → query DynamoDB
4. Store result in Redis
5. Return response

Flow:

```
Doctor Request
      │
      ▼
Redis Cache
  │        │
 HIT      MISS
  │        │
  ▼        ▼
Return   DynamoDB
           │
           ▼
        Update Cache
           │
           ▼
         Return
```

# 10. API Endpoints

## Create Patient

POST /patient

Response:

```
{
 "shn": "SHN-482193"
}
```

---

## Add Patient Data

POST /patient/{shn}/data

Request Body:

```
{
 "key": "blood_group",
 "value": "O+"
}
```

Action:

* Write to DynamoDB
* Invalidate Redis cache

---

## Get Patient Data

GET /patient/{shn}/data

Response:

```
{
 "shn": "SHN-482193",
 "data": {
   "blood_group": "O+",
   "age": 31
 }
}
```

# 15. Project Scope Limits

The following features are intentionally **out of scope** for this version:

* Blockchain integration
* OCR or document scanning
* Analytics dashboards
* Airflow pipelines
* PowerBI integration
* Authentication systems (OAuth / Cognito)

This project focuses on **clean architecture and core backend systems**.

The final system should demonstrate:

* A working cloud backend
* Scalable serverless architecture
* Fast data retrieval using Redis caching
* Proper API structure
* A simple usable frontend

The project should remain **small, clean, and easy to explain in interviews**.
